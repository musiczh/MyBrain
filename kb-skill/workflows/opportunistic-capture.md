# 机会性入库工作流

目标：在普通聊天中识别值得长期保存的用户想法、偏好、决策或经验，并在用户确认后写入本地知识库。默认先询问确认，不自动入库。

## 适用场景

- 用户表达了长期偏好、原则、方法论或反复出现的判断标准。
- 用户给出了明确决策、复盘结论、踩坑经验、可复用流程或验收标准。
- 用户说出个人感悟、工作经验、研究发现或未来希望持续遵循的规则。
- 当前对话形成了可复用的高价值回答，适合沉淀成主题页或实体页。

## 不适用场景

- 临时情绪、闲聊、一次性安排、敏感隐私、账号密钥、未确认的个人信息。
- 用户只是要求即时回答，且内容没有长期复用价值。
- 用户明确表示不要保存、不要记住或只是临时讨论。

## 步骤

1. 先判断是否值得建议入库。只对长期价值较明确的内容触发，不要频繁打扰用户。

2. 向用户确认，简短说明建议保存的原因和拟保存内容：

   ```text
   这段内容像是一个长期可复用的经验/偏好，我可以帮你存入知识库。是否保存？
   拟保存：<保留用户原始表达或最小必要上下文>
   ```

3. 用户同意后，先搜索已有编译层，避免重复建页：

   ```bash
   kb search "<关键词>" --layer wiki --json
   ```

4. 写入原料层。必须保存用户原始完整表达；如果需要上下文，用 `--context` 记录，不要替代原文：

   ```bash
   kb ingest --type thought --text "<用户原始完整表达>" --context "<当前对话必要上下文>" --json
   ```

   如果内容更像资料片段或待归档记录，可使用 `--type note`。

5. 获取编译上下文：

   ```bash
   kb plan <raw_id> --json
   ```

6. 按价值决定编译深度：

   - 轻量偏好或短想法：可只写入 raw，并用 `kb compiled` 记录标签。
   - 可复用经验、方法论或决策：更新最相关实体页或主题页。
   - 对已有知识形成补充、反例或约束：写入对应页面并保留 source id。

7. 需要编译时，优先更新已有页面；确实没有承载页时再新建：

   ```bash
   kb topic upsert "<主题标题>" --body-file topic.md --source <raw_id> --json
   kb entity upsert "<实体标题>" --body-file entity.md --source <raw_id> --json
   kb link <page_id_a> <page_id_b> --json
   kb compiled <raw_id> --tag "<标签>" --json
   ```

8. 向用户简要反馈保存结果：raw id、更新的页面、是否建立关联。

## 原则

- 默认先确认再入库；不要静默保存普通聊天内容。
- 保存 raw 时必须保留用户原始完整表达，不要只存整理后的观点。
- 编译层内容可以总结，但必须保留 source id，方便回溯。
- 对敏感、私密或不确定价值的内容宁可询问，不要自动处理。
