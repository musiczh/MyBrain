# 查询工作流

目标：把本地知识库作为用户已有知识积累的参考层，帮助 Agent 回答问题。知识库不是唯一数据来源；命中时融合进答案，未命中时正常基于当前对话、通用知识和必要外部信息回答。

## 步骤

1. 判断查询意图：

   - 用户明确要求“查知识库/第二大脑/我之前记过什么/根据我的知识库回答”：必须检索知识库。
   - 用户问题可能受个人积累、历史偏好、经验或过去决策影响：先轻量检索知识库作为个人上下文。
   - 与用户个人积累无关的普通事实问题：不强制使用知识库。

2. 先读索引或搜索编译层：

   ```bash
   kb index
   kb search "<用户问题关键词>" --layer wiki --json
   ```

3. 根据命中读取相关 Wiki 页：

   ```bash
   kb get <page_id> --json
   ```

4. 如果 Wiki 页的结构化知识不足、需要核验原文、或需要引用来源，再回溯原料：

   ```bash
   kb get-raw <raw_id> --json
   ```

5. 你用 LLM 综合回答用户，并明确区分：

   - 知识库已有结论。
   - 当前对话提供的信息。
   - 外部资料或通用知识。
   - 你的推断。

6. 如果知识库未命中，正常回答用户，不要编造知识库依据；可简要说明“知识库里没有找到相关记录”。

7. 如果回答本身有长期价值，询问用户是否归档；用户同意后用 `kb topic upsert` 或 `kb entity upsert` 回流为编译层页面。

## 原则

- 默认不直接全量扫描 `raw/`；先用 wiki 命中判断是否需要回溯原料。
- 搜索结果不足时换关键词，不要立刻读取所有原料。
- 回答需要可追溯时引用 page id 或 raw id。
- 不要只从知识库中已有内容作答；知识库只提供用户个人上下文和可追溯材料。
- 不要把知识库命中内容当作最新事实；对时效性问题仍要结合当前可用信息。
